1. Foundations & Architecture

Presented by: VABBER

Overall System Architecture

Technology Used

EdgeX Foundry: An open-source IoT middleware platform that handles data collection and processing at the edge.

Docker: A containerization platform used to package and run all our services, ensuring a consistent environment.

Kubernetes: A container orchestration platform that manages our application's lifecycle and automates scaling.

Errors Faced & Resolutions

Challenge 1: Understanding the API Structure

  • Symptom: Many of our initial `POST` requests to send data were failing, and we were unsure of the exact URL and JSON format required by the EdgeX V2 API.
  • Resolution: A major breakthrough came from exploring the EdgeX Foundry API documentation available through Swagger UI. By accessing the UI in the browser, we could see all available endpoints, their required parameters, and example JSON bodies. This was critical for debugging and correctly formatting our API calls.

Challenge 2: Port Conflicts

  • Symptom: EdgeX services failed to start, reporting address already in use for key ports.
  • Resolution: We identified that the ports were either in use by old processes or reserved by Windows. We fixed this by re-mapping the conflicting ports in `docker-compose.yml` (e.g., changing host port 59701 to 49701) and using docker container prune to remove old containers.

Challenge 3: Unhealthy Containers on Startup

  • Symptom: Core EdgeX services like core-data were stuck in a restart loop or reported as `unhealthy`.
  • Resolution: By checking the logs, we found missing configurations. We fixed this by adding critical environment variables like EDGEX_SECURITY_SECRET_STORE: "false" and ensuring `depends_on` clauses included condition: service_healthy to manage startup order.

Challenge 4: Corporate DNS / Network Issues

  • Symptom: Docker and Helm couldn't download required container images, failing with no such host errors.
  • Resolution: This was caused by a restrictive corporate DNS. We fixed it by manually configuring the DNS settings in both the Docker Desktop engine and the host Windows machine to use a public DNS server (8.8.8.8).

2. Data Generation & Ingestion

Presented by: Rohith Kumar D.

Implementation Details

Before EdgeX can accept data, the device must be formally registered. This is a mandatory, three-step process using the Core Metadata API on port 59881.

  1. Register Device Profile (health-monitor-profile.json): We first POST a JSON file that defines the device's "template"—what data it can send (e.g., "Heart-Rate" as an Int32).
  2. Register Device Service (health-monitor-service.json): We then POST a JSON file to register the service that will manage the device.
  3. Register Device (health-monitor-device.json): Finally, we POST a JSON file for the specific device instance, linking the profile and service together. Only after this step will Core Data accept events from the device.

Errors Faced & Resolutions

Challenge 1: Device Registration

  • Symptom: Our load-generator failed with errors like ApiVersion required, Id needs uuid, Origin required when trying to send events directly to Core Data.
  • Resolution: We realized EdgeX requires devices to be registered before posting events. To fix this, we created three files:
    • Profile – defined Heart Rate & Blood Pressure resources.
    • Service – logical service to manage the device.
    • Device – linked the profile and service.
    After registering them via Metadata (:59881/api/v2/... ), we were able to send events successfully to Core Data (:59880/api/v2/event/{profileName}/{deviceName}/{sourceName}).

Challenge 2: Unstable WSL Networking & Connection Failures

  • Symptom: The script still failed with connection errors (Timeout, Could not resolve host). The "WSL distro terminated abruptly" error was a key clue.
  • Resolution: We proved the host-to-container network was unstable. The final, robust solution was to build the `load-generator` directly into the `docker-compose-no-secty.yml` file. This allowed it to run inside the stable Docker network, bypassing the faulty WSL network bridge.

3. Overload Simulation & Monitoring

Presented by: Kalyan Kumar Reddy C

Technology Used

Prometheus: A time-series database that "scrapes" (collects) metrics from our Kubernetes pods, most importantly CPU utilization.

Implementation Details

The system overload is simulated by the load-generator service we added to our Docker Compose file. It runs a continuous loop, sending a high volume of data to the edgex-core-data service. This sustained load stresses our target application. In parallel, our Prometheus stack, installed via Helm, constantly collects performance data from the application pod.

Errors Faced & Resolutions

Challenge 1: HPA Metric Status was "<unknown>"

  • Symptom: The command kubectl get hpa showed <unknown>/500m for the TARGETS column, indicating the HPA could not fetch the custom metric.
  • Resolution: This was a multi-step fix. We first installed the Prometheus Adapter. Then, we corrected its configuration via Helm to point to the correct Prometheus service URL, fixing a "no such host" error. Finally, we simplified the adapter's rule in custom-metrics-values.yaml to correctly process the custom _avg metric from your application.

Challenge 2: Missing Scrape Job

  • Symptom: After patching, the new scrape pool local-target-app-direct did not appear in the Prometheus Targets page.
  • Resolution: Verified that the Secret additional-scrape-configs was mounted correctly and reapplied Prometheus manifests with proper additionalScrapeConfigs reference.

4. Visualization & Latency-Aware Autoscaling in Action

Presented by: Niharika

Technology Used

Grafana: The visualization platform where we built a dashboard to display live metrics from Prometheus.

Kubernetes HPA: The Horizontal Pod Autoscaler, configured to scale our application based on latency.

The Latency-Aware Autoscaling Event

Our implementation directly follows the advanced model proposed in the research paper. Instead of scaling on a crude metric like CPU, we scale based on the actual user experience: response time.

Our Grafana dashboard visualizes the entire feedback loop. The video shows:

  1. The "Average Response Latency (ms)" graph spikes past our 200ms target due to the simulated load.
  2. The Kubernetes HPA, monitoring this custom latency metric via Prometheus, detects the service degradation and triggers a scaling event.
  3. The terminal shows new pods being created.
  4. The "Active Pod Count" graph on the dashboard jumps from 1 to 4.
  5. As the new pods share the work, the average latency drops back below our target, demonstrating successful, user-centric load balancing.

Errors Faced & Resolutions

Challenge 1: Monitoring Pod Crashing

  • Symptom: A monitoring pod called prometheus-node-exporter was stuck in a CrashLoopBackOff state.
  • Resolution: This pod requires host-level permissions that Docker Desktop restricts. Since it wasn't critical, we reinstalled the Prometheus Helm chart with the component disabled via the flag --set prometheus-node-exporter.enabled=false.

Challenge 2: Grafana Panel Showed "No Data"

  • Symptom: The "Average Request Latency" panel in Grafana displayed "No Data", and running the query directly in Prometheus returned an "Empty query result".
  • Resolution: We identified that the application was running but not receiving any HTTP traffic. The solution was to use kubectl port-forward and a PowerShell script to send repeated POST requests to the application, which caused it to generate the required latency metrics.

5. Future Integration: The DRL Model Maker

Presented by: Aditya Suresh

Deep Reinforcement Learning for Edge-based Healthcare Prediction

We use a two-stage DRL process:

01

Supervised Warm-start (NOT DRL)

  • Uses Focal Loss to handle imbalance.
  • Gives the network a reasonable starting point.
  • Without this, DRL starts from random guessing → very unstable.
02

Policy Gradient (REINFORCE) Fine-tuning [DRL]

  • The actual DRL stage.
  • Agent predicts disease from sensor inputs.

Reward System:

  • ✅ Correct → positive reward.
  • ✅ Rare/critical disease → extra reward.
  • ❌ Wrong → penalty.
  • Learns policies that maximize long-term classification accuracy.

6. Future Integration: Federated Learning

Presented by: Himanshu M

A key challenge in healthcare is training AI models on diverse datasets without violating patient privacy. The planned future implementation to solve this is Federated Learning.

1

Local Training

Each edge node (e.g., a hospital's server) uses its own private patient data to locally train and improve its copy of the DRL model.

2

Share Anonymous Updates

Only the generalized, anonymous model updates (mathematical weights and biases) are sent to a central aggregation server. No private patient data ever leaves the edge node.

3

Aggregate & Distribute

The central server combines the updates from all edge nodes to create an improved "global model." This smarter global model is then sent back to all the edge nodes.

7. Future Work: The Core Transition

Our primary goal is to evolve from a reactive, system-based scaling model to a proactive, user-centric one.

⚙️Current: CPU-Based Scaling

  • Reactive: Scales only when the server's CPU is already high. By this point, users may already be experiencing delays.
  • Indirect Metric: High CPU usage does not always correlate with poor user experience (high latency), leading to inefficient scaling.
  • System-Centric: Focuses on the health of the server, not the quality of service delivered to the end-user.
→

⏱️Proposed: Latency-Aware Scaling

  • Proactive: Scales based on actual application response times, directly measuring the user's experience.
  • Direct Metric: Targets the most critical performance indicator for real-time systems, ensuring a consistently smooth and responsive service.
  • User-Centric: Guarantees more efficient real-time performance by scaling to meet user needs *before* the system becomes overloaded.
